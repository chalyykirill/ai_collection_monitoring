# Demo Guide

## Цель демо

Показать, что агент анализирует мониторинг Collection, отличает ожидаемые
события от потенциальных инцидентов и предлагает проверки на основании
документации и результатов диагностических tools.

## Команда запуска

```bash
.venv/bin/python run_full_demo.py
```

Полный запуск последовательно генерирует данные, алерты, комментарии,
расследования, финальное резюме, графики и HTML-отчет.

## Что открыть

```text
outputs/runs/demo_run/human_report.html
```

## Как рассказывать демо за 5 минут

### Шаг 1. Показать итоговое резюме

Начните с верхней части отчета. Обратите внимание на разделение результатов:

- 2 expected events;
- 4 potential incidents;
- отдельные гипотезы первопричин и приоритетные проверки.

Главная мысль: критический общий статус не заставляет агента ошибочно
эскалировать ожидаемые события.

### Шаг 2. Показать expected event: bank_unavailable_day

Откройте группу `bank_unavailable_day`.

Главная мысль: объем упал до нуля и расчетный алерт имеет статус `critical`,
но RAG-контекст и календарный признак подтверждают ЕДН. Агент объясняет, что
отклонение не является инцидентом, и рекомендует только проверить
восстановление потока.

### Шаг 3. Показать expected process feature: credit_card_batch_inflow

Откройте группу `credit_card_batch_inflow`.

Главная мысль: рост `cnt_clients` и `product_share` объяснен ожидаемым
пакетным поступлением портфеля `credit_card`. Агент рекомендует
контролировать завершение загрузки и анализировать метрики с учетом
продуктового микса.

### Шаг 4. Показать incident: empty_feature_vector

Откройте группу `empty_feature_vector`.

Главная мысль: агент связывает рост `empty_feature_vector_share` и
`feature_missing_rate` со сдвигом score, выбирает проверки качества данных и
формирует проверяемую гипотезу о проблеме в источниках признаков.

### Шаг 5. Показать incident: optimizer_mass_reject

Откройте группу `optimizer_mass_reject`.

Главная мысль: агент проверяет `reject_share`, communication mix и объем.
Результаты показывают проблему поведения оптимизатора, а не общее изменение
входящего потока.

### Шаг 6. Показать agent workflow

В блоке Investigation results покажите последовательность:

1. Planner выбирает tools из доступного списка.
2. Python проверяет whitelist и выполняет расчеты.
3. Tool results становятся evidence.
4. Summarizer формирует root cause hypothesis и recommended actions.

## Что сказать про RAG

Документация хранится в `docs/` и имитирует базу знаний процесса. Retrieval
ищет релевантные фрагменты только в `docs/common/` и каталоге конкретного
процесса. Найденные источники сохраняются в:

```text
outputs/runs/demo_run/retrieved_contexts.json
```

В отчете для каждой группы видны источник, score и текстовый фрагмент.

## Что сказать про безопасность

LLM не выполняет код. Она возвращает JSON-план с названиями tools. Python
принимает только инструменты из whitelist, удаляет дубли, ограничивает план
пятью вызовами и самостоятельно выполняет диагностические функции.

Все LLM-ответы валидируются через Pydantic. Дополнительно используются JSON
repair, language repair и deterministic final audit.

## Что сказать про ограничения

Данные синтетические и агрегированные, но шесть сценариев отражают
реалистичные задачи мониторинга Collection. MVP не подключен к промышленным
источникам, Confluence и scheduler.
